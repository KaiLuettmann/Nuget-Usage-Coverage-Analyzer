# NuGet Usage Coverage Analyzer

Analyzes which NuGet packages of a .NET solution are actually captured in test coverage data.
The tool reads a Cobertura XML coverage file and correlates it with the actual API usage sites
in production code via Roslyn semantic analysis.

## Prerequisites

- .NET 10 SDK (to run the tool)
- The solution being analyzed must be fully restored (`dotnet restore`)
- A Cobertura XML coverage file (e.g. generated by `dotnet test --collect:"XPlat Code Coverage"`)

Supported target frameworks of the analyzed solution: `net6.0`, `net7.0`, `net8.0`, `net9.0`,
`net10.0` (each also with `-windows`), `netstandard1.6`, `netstandard2.0`, `netstandard2.1`.
Additional TFMs are accepted as a fallback.

## Workflow

The analysis runs in four phases:

### Phase 0 — Package Audit

The tool reads all `.csproj` files from the `.sln`, evaluates their `obj/project.assets.json`
and determines all **direct** NuGet dependencies of the solution. Packages marked via
configuration as test, build, or internal packages are filtered out and appear as `SKIP` in the
report.

> **Note:** Without a prior `dotnet restore` no `project.assets.json` files exist and the
> analysis produces no results.

### Phase 1 — Load Coverage

The Cobertura XML file is read. For each source file the tool records which lines are
instrumented (coverable) and which of those are covered by tests.

### Phase 2 — Scan Source Files

1. All `.cs` files in the repo are collected (test files are excluded via the
   `sourceExcludePattern` regex).
2. A Roslyn compilation is built with the NuGet DLLs added as metadata references.
3. Every identifier in the syntax tree is semantically resolved. The containing assembly is
   mapped to a catalogue entry (NuGet package) using two strategies:
   - **Strategy 1 – Direct match:** The assembly is present in the `assemblyToPackage` map
     from `project.assets.json`.
   - **Strategy 2 – Prefix fallback:** The assembly name starts with a package name from the
     catalogue (e.g. `Microsoft.AspNetCore.Mvc.Core` → `Microsoft.AspNetCore.Mvc`). The
     longest match wins.
4. Each usage site (file + line) is correlated with the coverage data from Phase 1.

Detected usage kinds (`Kind`):

| Kind            | Example                         |
|-----------------|---------------------------------|
| `ObjectCreation`| `new MyService()`               |
| `Attribute`     | `[Authorize]`                   |
| `MethodCall`    | `logger.LogInformation(...)`    |
| `TypeRef`       | Type as parameter or return type|

### Phase 3 — Output Report

The report is written as a UTF-8 text file and contains:

- **Package Audit:** Overview of all packages (analyzed / skipped)
- **Summary table:** Per package: number of usages, coverable, covered, coverage % and status
- **Per-package detail:** All usage sites with file, line, kind and coverage symbol

Coverage status levels:

| Status              | Meaning                                               |
|---------------------|-------------------------------------------------------|
| `FULL_COVERED`      | 100 % of coverable usages are covered                 |
| `HIGH_COVERED`      | 80–99 % covered                                       |
| `MEDIUM_COVERED`    | 50–79 % covered                                       |
| `LOW_COVERED`       | 1–49 % covered                                        |
| `NOT_COVERED`       | Coverable usages exist but none are covered           |
| `NOT_INSTRUMENTED`  | Usages found but none are coverable                   |
| `NO_USAGES`         | No identifier matches found in production code        |

Symbols in the detail report: `C` = covered · `.` = coverable, not covered · `-` = not coverable

## Usage

```
dotnet run -- --solution <path\to.sln> --config <path\to\config.jsonc> [options]
```

### Required arguments

| Option               | Short | Description                              |
|----------------------|-------|------------------------------------------|
| `--solution <file>`  | `-s`  | Path to the `.sln` file                  |
| `--config <file>`    | `-c`  | Path to the `package-analysis-config.jsonc` |

### Optional arguments

| Option                 | Short | Default                                                       | Description                                           |
|------------------------|-------|---------------------------------------------------------------|-------------------------------------------------------|
| `--coverage <file>`    | `-x`  | `<Repo>\test-results\coverage-report\Cobertura.xml`       | Path to the Cobertura XML coverage file               |
| `--output <file>`      | `-o`  | `<Repo>\NuGetUsageCoverageAnalyzer\nuget-coverage-output.txt` | Output file                                           |
| `--verbose`            | `-v`  | `false`                                                       | Detailed diagnostics: DLL resolution, assembly map, counts |
| `--include-transitive` | —     | `false`                                                       | Include transitive (indirect) dependencies            |

### Example

```powershell
dotnet run `
    -- --solution D:\dev\MyProject\MyProject.sln `
       --config D:\dev\MyProject\NuGetUsageCoverageAnalyzer\package-analysis-config.jsonc `
       --coverage D:\dev\MyProject\test-results\coverage-report\Cobertura.xml
```

Console output:

```
Solution : D:\dev\MyProject\MyProject.sln
Config   : D:\dev\MyProject\NuGetUsageCoverageAnalyzer\package-analysis-config.jsonc
Output   : D:\dev\MyProject\NuGetUsageCoverageAnalyzer\nuget-coverage-output.txt
```

## Configuration file

The `package-analysis-config.jsonc` controls which packages are excluded and which source files
count as production code.

```jsonc
{
  // Packages that match exactly are skipped
  "skipExact": [
    "xunit",
    "FluentAssertions",
    "NSubstitute",
    "Microsoft.NET.Test.Sdk"
    // ...
  ],

  // Packages whose name starts with one of these prefixes are skipped
  "skipPrefixes": [
    "runtime."   // RID-specific runtime packs
  ],

  // Internal packages that are also skipped (own libraries etc.)
  "skipInternalPrefixes": [
    "MyCompany.",
    "InternalLib."
  ],

  // Regex applied to every .cs file path.
  // Files that match are excluded as non-production code.
  "sourceExcludePattern": "\\\\(Test|AcceptanceTests)\\\\|\\.Test[\\\\.]"
}
```

### Skip categories

- **`skipExact`** — Test frameworks, analyzers, build tools, BCL meta-packages
- **`skipPrefixes`** — RID-specific runtime packs (`runtime.*`)
- **`skipInternalPrefixes`** — Own/internal libraries
- **`sourceExcludePattern`** — Regex to exclude test files from the source file scan

## Troubleshooting

**`NO_USAGES` for a package that is definitely used**

1. Enable `--verbose` — shows the assembly map and usage count per package.
2. Check whether the package is listed in `skipExact` / `skipPrefixes` / `skipInternalPrefixes`.
3. Make sure `dotnet restore` has been run and `project.assets.json` is up to date.
4. For umbrella packages (e.g. `Swashbuckle.AspNetCore`): the prefix fallback only applies when
   the package name is a prefix of the assembly name.

**Coverage file cannot be parsed**

If the coverage file is missing or in an invalid format, the tool continues and treats all usage
sites as not instrumented. Packages with usages will show status `NOT_INSTRUMENTED`; packages
with no usages remain `NO_USAGES`.

Generating coverage with ReportGenerator:

```powershell
dotnet test --collect:"XPlat Code Coverage" --results-directory test-results

dotnet tool run reportgenerator `
    -reports:"test-results\**\coverage.cobertura.xml" `
    -targetdir:"test-results\coverage-report" `
    -reporttypes:Cobertura
```
